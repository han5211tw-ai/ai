## 十一、客戶待建檔高標準匹配機制

### 設計原則
- **ERP 為唯一正式主檔來源**
- **嚴格禁止模糊姓名匹配**
- **僅當手機號碼完全一致時才自動對應**
- **所有操作保留審計紀錄**

### 系統分層

| 層級 | 資料表 | 說明 |
|------|--------|------|
| 正式主檔 | `customer_master` | 每日從 ERP 匯入，唯一正式主檔（含 customer_id） |
| 待建檔區 | `customer_staging` | 前線送出的新客戶資料（無 customer_id） |

### 資料表結構

#### customer_master（ERP 主檔）
| 欄位 | 說明 |
|------|------|
| customer_id | ERP 客戶編號（唯一） |
| short_name | 客戶簡稱 |
| mobile | 正規化後的手機號 |
| mobile_raw | 原始手機號 |
| import_date | 匯入日期 |

#### customer_staging（待建檔）
| 欄位 | 說明 |
|------|------|
| status | pending/synced/needs_review/rejected |
| match_reason | 匹配原因 |
| matched_at | 匹配時間 |
| erp_customer_id | 對應的 ERP 客戶編號 |
| audit_log | 審計日誌（JSON 陣列） |

### 匹配規則（高標準）

**自動匹配條件**：
1. 手機號碼正規化後**完全一致**
2. 在 master 中**僅找到 1 筆**符合資料
3. 禁止模糊姓名匹配作為自動條件

**手機號正規化規則**：
- 去除所有空白和 `-` 符號
- `+886` 前綴改為 `0`
- 確保為 10 碼數字（09xxxxxxxx）

**匹配結果處理**：
| 匹配結果 | 處理方式 |
|----------|----------|
| 找到 0 筆 | 保持 `pending` |
| 找到 1 筆 | 自動匹配 → `synced` |
| 找到 >1 筆 | 標記 `needs_review`（需人工審核） |
| 無有效手機 | 保持 `pending` |

### 自動匹配任務

**執行時間**：每日 10:46（接續在客戶模組 10:45 後）

**Crontab 設定**：
```
# [客戶待建檔自動匹配] 每天 10:46（接續在客戶模組後）
46 10 * * * /opt/homebrew/bin/python3 /Users/aiserver/srv/parser/match_customer_staging.py >> /Users/aiserver/srv/logs/match_staging.log 2>&1
```

**腳本位置**：`/Users/aiserver/srv/parser/match_customer_staging.py`

### API 端點

```
GET    /api/staging/customer/needs-review          # 獲取需人工審核列表
POST   /api/staging/customer/{id}/manual-match     # 人工匹配
GET    /api/customer-master/search?keyword=        # 搜尋 Master 客戶
POST   /api/staging/customer/run-match             # 手動執行自動匹配
```

### 狀態流向

```
pending ──自動匹配成功──→ synced
   │
   ├──找到多筆或無手機──→ needs_review
   │                         │
   │                         └──人工審核──→ synced
   │
   └──無法匹配──────────→ pending（等待 ERP 建立）
```

### 人工審核流程

1. **進入管理頁面**：`staging_management.html`

5. **確認匹配**
6. **系統自動**：
   - 更新 staging status = 'synced'
   - 回填 erp_customer_id
   - 更新對應需求單 customer_code
   - 記錄審計日誌

## 二十二、防暴力破解機制（2026-02-28 上午）

### 功能規格
- **追蹤來源 IP**：記錄每個登入嘗試的 IP 位址
- **失敗次數限制**：同一 IP 連續失敗 5 次後鎖定 15 分鐘
- **鎖定期間拒絕**：即使密碼正確也無法登入
- **成功登入重置**：登入成功後自動清除失敗記錄

### 資料表結構
```sql
CREATE TABLE login_attempts (
    ip_address TEXT PRIMARY KEY,
    failed_count INTEGER DEFAULT 0,
    locked_until DATETIME,
    last_attempt DATETIME DEFAULT CURRENT_TIMESTAMP
);
```

### API 實作
- `get_client_ip()`：獲取真實 IP（支援反向代理）
- `check_ip_locked(ip)`：檢查 IP 鎖定狀態
- `record_failed_login(ip)`：記錄失敗並檢查是否觸發鎖定
- `reset_login_attempts(ip)`：登入成功時重置記錄
