# 2026-03-19 - 已採購通知系統修復

## 問題描述

**員工提交的「請購」需求（ID:15097 駿程 監控專用 接線防水盒）** 在老闆點擊「已採購」按鈕後，狀態雖然變成「已採購」，但**沒有發送到電腦舖工作群組**。

與此同時，**「調撥」需求（ID:15100 廣鐸 HDMI 轉 VGA 接線）** 的已經正常發送到工作群組。

## 問題分析

### 原始程式碼邏輯

在 `purchase_need()` 函數（第 3496-3575 行）中，程式碼有狀態檢查：

```python
if row['status'] != '待處理':
    return jsonify({'success': False, 'message': '該筆資料無法標記已採購'})
```

**問題根源**：
1. 如果狀態已經是「已採購」，程式碼直接返回錯誤
2. **連嘗試發送通知的機會都沒有**，程式碼根本不會執行到發送 Telegram 通知的部分
3. 結果是「ID 15097 變成已採購時，系統完全沒有記錄到通知日誌」

### 為什麼會發生？

- **ID 15097** 原本狀態就不是「待處理」（可能是重複點擊、重新提交，或從其他狀態轉換）
- 當狀態已經是「已採購」時，系統直接拒絕執行，返回 `{'success': False, 'message': '該筆資料無法標記已採購'}`
- **連嘗試發送通知的機會都沒有**，程式碼根本不會執行到發送 Telegram 通知的部分

### 資料庫證據

```sql
SELECT * FROM notification_logs WHERE related_record_id = 15097;
-- 結果：NO RECORDS FOUND - ID 15097 沒有發送通知到 notification_logs
```

## 修復方案

### 修改程式碼

**修改位置**：`/Users/aiserver/.openclaw/workspace/dashboard-site/app.py` 第 3530-3577 行

**修改內容**：
1. **移除狀態檢查**：將 `if row['status'] != '待處理'` 改為只在狀態是「待處理」時才更新狀態
2. **保持發送通知**：無論狀態為何，**一律發送通知到工作群組**
3. **狀態顯示優化**：如果狀態已經是「已採購」，在通知中顯示「再次已採購」

### 修改後邏輯

```python
# 標記為已採購（如果狀態已經是已採購，就不更新，但仍發送通知）
now = datetime.now().strftime('%Y-%m-%d %H:%M:%S')
if row['status'] == '待處理':
    cursor.execute("""
        UPDATE needs
        SET status = '已採購', processed_at = ?
        WHERE id = ?
    """, (now, need_id))
    conn.commit()

# 發送 Telegram 通知到工作群組（背景執行，不阻塞回應）
try:
    old_status = row['status']
    status_text = "再次已採購" if old_status == '已採購' else "已採購"
    
    telegram_msg = f"""✅ <b>已採購通知</b>

📅 日期：{need_date}
👤 填表人：{original_requester}
📍 部門：{department}
📦 產品：{item_name}
🔢 數量：{quantity} 個
📌 狀態：{status_text}

請於到貨後至營運系統首頁結案"""

    TELEGRAM_GROUP_CHAT_ID = "-5232179482"
    import threading
    t = threading.Thread(
        target=send_telegram_notification,
        args=(telegram_msg, TELEGRAM_GROUP_CHAT_ID, '已採購通知', need_id, 'needs')
    )
    t.start()
except Exception as e:
    print(f"已採購 Telegram 通知背景執行緒啟動失敗：{e}")

return jsonify({'success': True})
```
